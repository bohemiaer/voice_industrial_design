# 工业设计脑暴工具需求文档 v1

日期：2026-06-25

## 1. 概述

本产品是一款面向企业工业设计师的 Web 端脑暴画布，用于在项目 kickoff 之后，把模糊需求快速发散成多个造型语言差异明显的概念方向。

产品核心不是“输入一句出一张图”，而是“输入一个意图，长出一棵概念树”。用户可以通过语音、文字或画布操作表达需求、补充限制、选择方向、刷新分支、撤回操作或进行只读问答；系统将这些来源归一成标准意图，并在需要生成时持续生成分支化草图节点，帮助设计师保留完整的概念演化路径。

第一阶段核心用户为企业工业设计师。后续可扩展到共创 workshop 场景，但不纳入 v1 范围。

## 1.1 v1 统一口径

1. 用户输入来源包括语音、文字和画布操作。
2. 不同来源先由 Orchestrator 归一成标准意图，再路由到闲聊、树操作或生成链路。
3. 生成类动作统一为：
   - `diverge`：沿当前节点新增一组子节点
   - `refresh`：刷新当前节点下最近一次生成出来的那组子节点
4. 树操作统一为：
   - `delete`：删除目标节点及其整棵子树
   - `undo`：撤回最近一次 `diverge / refresh / delete`
   - `redo`：恢复最近一次被 `undo` 的操作
5. `delete / undo / redo` 由 Orchestrator 直接执行，不进入脑暴和生图链路。
6. 闲聊、帮助、状态查询和节点解释走独立 `chat` 分支，只读回答，不进入脑暴和生图链路，不写树结构。
7. 主链路采用直执行模型，不设置额外审批卡或中间等待态。
8. `refresh` 若未明确数量，默认沿用被刷新生成组的原始数量；`diverge` 若未明确数量，默认生成 3 个子方向。
9. 不能用简单正则从原始文本覆盖 LLM 对数量的语义理解，例如“设计一个咖啡壶”不等于“只生成一个方向”。
10. 根节点不可删除；删除当前节点后，当前节点回退到父节点，没有父节点时回根节点。
11. 任意生成任务进行中时，新的生成输入应被阻止或排队，避免并发污染树状态；闲聊分支可以继续回答不依赖树写入的问题。

## 2. 目标与非目标

## 2.1 目标

1. 帮助工业设计师在 kickoff 阶段快速完成前期发散。
2. 让需求到图像方向的转换更快、更自然，减少手动整理和反复描述成本。
3. 通过树状结构保留方案的分支关系，让讨论过程更有结构。
4. 让 AI 输出不仅有图，还能解释每个方向为什么成立。
5. 让语音、文字和画布操作都能进入同一套标准意图链路。
6. 支持不改变画布的闲聊、帮助、状态查询和节点解释。

## 2.2 非目标

1. 不做高保真工业设计渲染或最终效果图工具。
2. 不做专业建模、CAD 或生产链路集成。
3. 不做多人协作和 workshop 联机能力。
4. 不做深度白板编辑、自由排版、框选编组和批量批注。
5. 不做大量参数滑杆和复杂控制面板，避免退化为通用生图工具。
6. 不设置额外审批卡作为 v1 主流程的一部分。

## 3. 核心用户与使用场景

## 3.1 核心用户

企业内的工业设计师，尤其是在新项目启动、方向探索、前期概念发散阶段需要快速生成多个造型方向的人。

## 3.2 核心场景

用户在项目 kickoff 之后，针对一个仍然模糊的产品需求，通过语音、文字或画布操作表达目标、场景、风格倾向和功能重点。AI 基于标准意图生成多个造型语言差异明显的概念草图分支。用户选择其中一个方向继续深入、刷新当前节点下最近一组结果，或撤回/重做树操作，逐步形成一棵概念演化树。过程中用户也可以询问当前节点含义、使用方式或画布状态，系统以只读方式回答，不改变树结构。

## 3.3 核心痛点

1. 前期发散速度慢，从语言到图像的转换成本高。
2. 设计讨论容易散，方案很多但缺少结构化关系。
3. 通用生图工具难以保留“为什么这么分叉”的演化过程。
4. 纯聊天工具不擅长表达父子关系、刷新版本和撤回重做。

## 4. 产品定位

一句话定位：

一个面向企业工业设计师的脑暴画布，让模糊需求沿着可视化分支快速长成多个造型方向。

差异化表述：

它不是“输入一句出一张图”的 AI 生图工具，而是“输入一个意图，长出一棵概念树”的工业设计前期发散工具。

## 5. 核心体验原则

1. 右边表达，左边生长：输入和理解发生在对话区，结果和演化发生在画布区。
2. 先发散，再收敛：前几轮草图更松、更强调方向差异，后续逐步更具体。
3. 结构先于数量：比起生成更多图，更重要的是用户始终知道当前位于哪条分支、这一轮发生了什么。
4. 图和解释并重：每个方向不仅有草图，也有简短解释支持判断。
5. 工业设计语境优先：提示、解释和方向命名都要贴近工业设计表达，而不是泛视觉创作语言。
6. 来源归一：语音、文字和画布操作都应落到同一套标准意图，不让不同入口产生不同业务语义。
7. 问答不扰动画布：闲聊和解释类输入只影响对话区，不隐式触发生成或树操作。

## 6. 信息架构

## 6.1 整体布局

页面采用双区结构：

1. 左侧为白色画布，承载树状概念演化结构。
2. 右侧为轻量对话区，承载输入记录、AI 回复、状态摘要和操作入口。

## 6.2 画布区

画布区用于展示从根节点开始不断生长的概念树。

1. 根节点表示初始需求。
2. 每一层节点表示 AI 基于某轮上下文生成的一组方向。
3. 节点之间以分支关系连接，形成清晰演化路径。
4. 每个节点必须有稳定可读的命名和会话内唯一数字序号。
5. 用户可通过画布点击、语音引用或文字引用选择任一节点作为下一轮操作对象。

## 6.3 对话区

对话区不是参数面板，而是思考对话空间，主要包括：

1. 用户输入记录，来源可以是语音转写、文字输入或画布操作摘要。
2. AI 的简短理解与反馈。
3. 当前轮次的输入提示，例如建议从比例、材质、部件、品牌气质等维度继续描述。
4. 当前操作状态提示，包括本轮识别到的意图类型、作用对象和预计结果。
5. 不改变画布的闲聊、帮助、状态查询和节点解释回复。

v1 对话区遵循“默认简洁、运行时展开、完成后折叠”的原则：

1. 工作台进入后默认只保留当前节点、最近摘要和主要操作入口。
2. 系统处于理解、生成等运行态时，实时展开反馈历史。
3. 本轮结束后，系统反馈折叠为一条摘要，例如“已基于节点 12 生成 3 个更轻薄的子方向”。

## 6.4 页面模块

v1 工作台固定由以下模块组成：

1. `TopBar`：显示会话标题、连接状态和当前任务是否运行中。
2. `CanvasWorkspace`：展示概念树、当前选中节点和最近新增结果高亮。
3. `CurrentTargetBanner`：显示当前操作对象的名称、数字序号和层内位置。
4. `ConversationPanel`：展示用户输入、运行中反馈和完成摘要。
5. `InputBar`：统一承载语音和文字输入。
6. `CanvasActionControls`：承载刷新、删除、撤回、重做等画布操作。
7. `IntentStatusCard`：在理解或执行阶段展示操作类型、目标节点和预计结果。

## 7. 节点模型

## 7.1 节点默认展示

每个节点默认显示：

1. 一张草图感概念图。
2. 一个简短且稳定的方向名称。
3. 一个节点独有的数字序号，例如 `12`。
4. 一个用于当前层理解的层内序号，例如“当前层第 2 个方向”。

## 7.2 节点展开信息

用户选中或展开节点后，显示一段简短解释：

1. 设计意图摘要。
2. 造型语言关键词。
3. 当前节点在树中的相对位置。
4. 可继续发散或刷新该节点的操作入口。

解释长度应控制在短段落级别，避免节点信息过重。

## 8. 核心交互模型

## 8.1 基本循环

产品的基本工作流为：

表达意图 -> 生成多分支 -> 阅读解释 -> 选择路径 -> 继续表达 -> 树继续生长

每一轮都应让用户清楚知道：

1. 当前选中了哪个节点。
2. 本轮输入作用于哪一层。
3. 本轮是新增分支、刷新当前节点下最近一组结果、删除、撤回还是重做。

## 8.2 输入来源

v1 支持三类输入来源：

1. 语音输入：用户说出需求、修改或操作命令，系统先转写为文本。
2. 文字输入：用户直接输入需求、修改或操作命令。
3. 画布操作：用户点击节点或按钮，例如刷新当前节点、删除节点、撤回、重做。

所有来源进入 Orchestrator 后都被归一为标准意图：

```ts
type StandardTurnIntent = {
  userIntentText: string;
  intentKind: "chat" | "generation" | "tree_op";
  requestedAction?: "diverge" | "refresh" | "delete" | "undo" | "redo";
  chatType?: "casual" | "help" | "status" | "explain_node" | "explain_canvas";
  targetNodeId: string;
  source: "voice" | "text" | "canvas";
};
```

`source` 用于日志和问题定位，不作为 Brainstorm Assistant 的核心创意输入。
`intentKind` 用于 Orchestrator 路由：`chat` 进入只读回复链路，`tree_op` 进入确定性树操作，`generation` 才进入 Brainstorm Assistant 和生图链路。
`chatType` 只在 `intentKind = "chat"` 时使用，用于选择闲聊、帮助、状态或解释回复风格。

## 8.3 生成类动作

### A. diverge

用户希望沿当前目标节点新增一组子方向。

示例：

“围绕这个咖啡壶先发散三个方向。”

结果：

在当前目标节点下新增一组子节点，不覆盖已有结果。

### B. refresh

用户希望刷新当前目标节点下最近一次生成出来的那组子节点。

示例：

“这组换一版，更轻薄一点。”

结果：

生成一个新的子节点组，旧组标记为 superseded，用于撤回和回放。

## 8.4 树操作

### delete

删除目标节点及其整棵子树。根节点不可删除。

### undo

撤回最近一次可撤回的 `diverge / refresh / delete`。

### redo

恢复最近一次被 `undo` 的操作。一旦执行新的 `diverge / refresh / delete`，redo 语义失效。

## 8.5 闲聊与只读问答

闲聊分支用于处理不要求改变画布的输入，包括：

1. 普通闲聊，例如“你好”“你是谁”。
2. 使用帮助，例如“我应该怎么继续”“这个工具怎么用”。
3. 状态查询，例如“现在选中的是哪个节点”“刚才生成了几个方向”。
4. 节点解释，例如“解释一下当前节点”“这几个方向有什么区别”。
5. 明确不想生成的讨论，例如“我想先聊聊，不要生成”。

闲聊分支的执行规则：

1. Orchestrator 识别为 `intentKind = "chat"` 后，调用 Chat Assistant。
2. Chat Assistant 只能读取会话目标、当前选中节点、可见节点摘要和近期对话摘要。
3. Chat Assistant 只返回 `assistantReply`。
4. 系统只写入 `messages`，不创建 `generation_tasks`，不创建 `branch_tasks`，不写入 `tree_operations`，不修改 `tree_nodes`。
5. 如果用户在闲聊中进一步提出明确生成或树操作意图，下一轮再由 Orchestrator 重新归一和路由。

## 8.6 意图识别策略

第一版不要求用户手动切换模式。系统优先根据标准意图和当前选中节点自动判断意图类型。

为降低误判成本，系统需要在对话区清晰显示：

1. 已识别的操作类型。
2. 作用对象节点。
3. 本轮将执行的结果形态，例如“新增 3 个子方向”“刷新当前节点下最近一组结果”或“只回答，不改变画布”。

如果用户发现理解错误，应能继续用自然语言纠正，或通过撤回恢复上一状态。

## 8.7 关键页面流程

### 首轮生成流程

1. 用户进入工作台后，通过语音或文字描述第一轮需求。
2. Orchestrator 归一为标准意图。
3. Brainstorm Assistant 理解需求并生成 3 个默认方向，除非用户明确要求其他合法数量。
4. 生图助理为每个方向生成草图。
5. 左侧生成根节点和首层分支，右侧展示完成摘要。

### 继续发散流程

1. 用户选择一个节点，或通过语音/文字引用一个节点。
2. 用户表达继续发散意图。
3. 系统沿目标节点新增一组子节点。
4. 本轮完成后，高亮新增节点组。

### 刷新流程

1. 用户选中某节点，触发刷新。
2. 系统定位该节点下最近一次生成组。
3. Brainstorm Assistant 生成新的方向 brief。
4. 系统写入新生成组，并将旧生成组标记为 superseded。

### 撤回/重做流程

1. 用户通过语音、文字或画布按钮触发撤回/重做。
2. Orchestrator 查找最近可撤回或可重做操作。
3. 系统更新节点可见性并写入新的树操作记录。
4. 对话区展示完成摘要。

### 闲聊/解释流程

1. 用户通过语音或文字提出不改变画布的问题。
2. Orchestrator 归一为 `intentKind = "chat"`，并标注 `chatType`。
3. Chat Assistant 基于当前会话、当前节点和必要的可见节点摘要生成回复。
4. 对话区展示回复，画布保持不变。

## 9. 输出风格节奏

草图输出应随分支深度逐步变化：

1. 前几轮偏松散、概念化、方向差异明显。
2. 中间轮次开始呈现更清楚的轮廓关系和工业设计特征。
3. 后续深入节点可逐步变得更具体，便于团队继续讨论和筛选。

v1 不追求高保真效果图，应优先保证方向差异和概念表达清晰。

## 10. 提示系统

当用户不确定如何继续描述时，系统应提供少量发散提示。提示内容应贴近工业设计思维，优先从以下维度引导：

1. 比例关系。
2. 材质与 CMF 倾向。
3. 关键部件和交互区域。
4. 品牌气质与产品语义。
5. 使用场景与目标人群感受。

提示应短小、可直接接话，不应变成长篇教程。

## 11. v1 功能范围

## 11.1 必做

1. Web 端脑暴画布。
2. 语音和文字输入。
3. 画布节点选择和基础操作入口。
4. 基于节点的多分支结果生成。
5. 节点草图与简短解释展示。
6. 每个节点稳定命名，并支持语音/文字引用。
7. `diverge / refresh / delete / undo / redo`。
8. 闲聊、帮助、状态查询和节点解释。
9. 少量发散提示能力。
10. 结果图片导出。

## 11.2 不做

1. 多人协作。
2. 高保真渲染与精细后处理。
3. 专业建模软件衔接。
4. 深度白板编辑能力。
5. 大量参数化控制。
6. 视觉 review 主链路能力。
7. 额外审批卡和中间等待态主流程。

## 12. 端到端用户流程

1. 用户进入页面，看到左侧画布和右侧对话区。
2. 系统提示用户描述项目需求、目标用户、场景和期望气质。
3. 用户通过语音或文字输入 kickoff 阶段的初始需求。
4. 系统归一标准意图，并在对话区显示理解摘要。
5. AI 在根节点下生成默认 3 个第一层方向，每个方向包含草图和解释。
6. 用户横向比较当前层结果。
7. 用户选择继续发散、刷新当前节点下最近一组结果、删除节点、撤回、重做，或询问当前节点和画布状态。
8. 系统识别本轮意图，在对话区显示操作类型和作用对象。
9. 如果是生成或树操作，画布更新，树结构继续生长或回退；如果是闲聊，画布保持不变。
10. 用户重复该循环，直到得到足够清晰的概念方向。

## 13. 成功标准

1. 用户能在短时间内生成多个风格差异明显的初始方向。
2. 用户能理解并接受树状分支作为主要工作方式。
3. 用户能清楚分辨 `diverge`、`refresh`、`delete`、`undo`、`redo`。
4. 节点解释确实帮助用户做选择，而不是成为额外噪音。
5. 语音、文字和画布操作能得到一致的业务结果。
6. 系统不会把“设计一个产品”误解为“只生成一个方向”。
7. 闲聊、帮助和解释类输入不会误触发生成或树操作。
8. 用户认为它比通用文生图工具更适合工业设计 kickoff 发散。

## 14. 后续扩展方向

1. workshop 多人共创模式。
2. 节点对比、收藏和筛选机制。
3. 方向汇总为方案板或提案页。
4. 更强的工业设计语义模板与行业化提示。
5. 与下游建模或设计交付工具的连接。
6. 视觉 review 和自动重试建议。
