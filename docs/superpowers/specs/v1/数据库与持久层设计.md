# 工业设计脑暴工具：数据库与持久层设计 v1

日期：2026-06-25

关联文档：

- [需求文档](./需求文档.md)
- [Agent PRD](./agent-PRD.md)

## 1. 目标

本文档定义 v1 阶段的数据库结构、关键索引、事务边界和持久层职责，重点回答：

1. 哪些核心对象需要落表。
2. 哪些字段必须结构化，哪些字段可以用 JSONB 承载。
3. `diverge / refresh / delete / undo / redo` 如何被持久化。
4. 如何记录生成组，支持刷新、撤回和重做。
5. 闲聊、帮助、状态查询和节点解释如何只写消息、不写树结构。
6. 近期对话摘要如何作为记忆压缩结果持久化。
7. 持久层应该按什么边界拆 repository。

本文档是 v1 设计口径，不保留旧的审批式中间链路和旧 action 名。

## 2. 总体原则

1. 以 PostgreSQL 为主存储。
2. 以关系型字段承载查询主路径，以 JSONB 承载可变结构。
3. 所有会修改树结构的操作都必须形成 `tree_operations` 记录。
4. 任何会影响多个节点和任务状态的写操作，必须在同一个事务里完成。
5. `refresh` 不物理覆盖旧节点，而是通过生成组版本切换和 superseded 标记实现。
6. `undo / redo` 都需要被持久化建模。
7. 数据库不保存审批式中间态作为主链路字段。
8. 用户输入来源可以是语音、文字或画布操作；业务层统一为标准意图后再落任务、操作记录或只读消息。
9. `chat` 分支只写入 `messages`，不创建 `generation_tasks`、`branch_tasks` 或 `tree_operations`。
10. `memory_summary` 是上下文压缩结果，只服务后续输入组装，不参与树写入。

## 3. 表清单

v1 最少落以下 6 张核心表：

1. `sessions`
2. `messages`
3. `tree_nodes`
4. `generation_tasks`
5. `branch_tasks`
6. `tree_operations`

后续扩展表：

1. `image_assets`
2. `node_reviews`

v1 不要求先落后续扩展表，只需在设计中保留接口预期。

节点下方的 3 个推荐发散方向属于节点展示字段，不作为独立核心表落库，也不作为独立树节点分配编号。v1 由 Brainstorm Assistant 在每个 direction brief 中强制输出 3 条，并随 `branch_tasks.brief_payload` 和 `tree_nodes.suggested_followups` 保存；用户点击推荐方向后，只有最终发送的输入会按普通消息和后续标准意图链路持久化。

## 4. 表设计

## 4.1 sessions

用途：

1. 表示一个完整脑暴会话。
2. 持有当前选中节点、最近执行目标和节点编号分配游标。
3. 记录会话所有权，便于账号隔离。

关键字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | `uuid` | 主键 |
| `owner_user_id` | `text` | 会话所属用户 |
| `title` | `text` | 会话标题 |
| `goal` | `text` | 会话初始目标 |
| `product_domain` | `text` | 固定为 `industrial_design` |
| `current_selected_node_id` | `uuid nullable` | 当前选中节点；为空表示 root |
| `last_executed_target_node_id` | `uuid nullable` | 最近一次执行树操作的目标节点 |
| `last_mentioned_node_id` | `uuid nullable` | 最近一次被用户或系统提及的节点 |
| `next_public_node_number` | `integer` | 下一个可分配数字序号 |
| `created_at` | `timestamptz` | 创建时间 |
| `updated_at` | `timestamptz` | 更新时间 |

约束：

1. `next_public_node_number >= 1`
2. `product_domain = 'industrial_design'`

索引：

1. `idx_sessions_owner_updated_at`
2. `idx_sessions_updated_at`

## 4.2 messages

用途：

1. 保存用户输入、系统状态、AI 回复和操作摘要。
2. 为 Brainstorm Assistant 提供裁剪后的近期上下文。

关键字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | `uuid` | 主键 |
| `session_id` | `uuid` | 所属会话 |
| `task_id` | `uuid nullable` | 关联生成任务 |
| `role` | `text` | `user` / `assistant` / `system` |
| `kind` | `text` | `intent` / `transcript` / `status` / `summary` / `hint` / `chat` / `node_explanation` / `memory_summary` |
| `content` | `text` | 消息正文 |
| `created_at` | `timestamptz` | 创建时间 |

说明：

1. 语音输入的转写结果可保存为 `transcript`。
2. 文字输入和画布操作归一后的标准意图可保存为 `intent`。
3. `summary` 用于保存 AI 回复和操作完成摘要。
4. `chat` 用于保存不改变画布的普通闲聊、帮助和状态查询。
5. `node_explanation` 用于保存不改变画布的节点或画布解释。
6. 闲聊分支的消息 `task_id` 必须为空，因为它不对应生成任务。
7. `memory_summary` 用于保存 Memory Summarizer 的压缩结果，`role = system`，`task_id` 为空。
8. `memory_summary` 的 `content` 应保存可读短摘要；结构化记忆可在后续加 `metadata jsonb`，v1 可先将 JSON 字符串写入 `content`。

索引：

1. `idx_messages_session_created_at`
2. `idx_messages_task_id`

## 4.3 tree_nodes

用途：

1. 保存概念树节点。
2. 支持稳定命名、数字序号引用、生成组、刷新版本和撤回重做。

关键字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | `uuid` | 主键 |
| `session_id` | `uuid` | 所属会话 |
| `parent_node_id` | `uuid nullable` | 父节点；为空表示 root 下首层 |
| `created_from_task_id` | `uuid nullable` | 来源生成任务 |
| `child_group_id` | `uuid nullable` | 同一次生成操作产生的兄弟节点组 |
| `depth` | `integer` | 树深度 |
| `layer_ordinal` | `integer` | 父节点下稳定顺序 |
| `layer_version` | `integer` | 父节点下版本号 |
| `public_node_number` | `integer` | 会话内唯一数字序号 |
| `display_name` | `text` | 口头可说出的稳定名称 |
| `label` | `text` | 前端显示标签 |
| `voice_aliases` | `jsonb` | 语音/文字别名数组 |
| `intent_summary` | `text` | 方向摘要 |
| `form_language` | `jsonb` | 造型语言数组 |
| `user_need_response` | `jsonb` | 该方向回应的用户需求或约束 |
| `inspiration_hints` | `jsonb` | 该方向可参考的产品、场景或设计线索 |
| `suggested_followups` | `jsonb` | 该节点下方展示的 3 条推荐发散方向 |
| `variation_axis` | `text nullable` | 与同组兄弟方向的主要差异维度 |
| `prompt_intent` | `text nullable` | 生图核心意图 |
| `image_url` | `text nullable` | 草图 URL |
| `status` | `text` | `ready` / `failed` |
| `superseded_at` | `timestamptz nullable` | 被刷新、删除或撤回隐藏的时间 |
| `superseded_by_operation_id` | `uuid nullable` | 哪次操作隐藏了它 |
| `created_at` | `timestamptz` | 创建时间 |
| `updated_at` | `timestamptz` | 更新时间 |

约束：

1. 同一 `session_id` 下，`public_node_number` 唯一。
2. 同一 `parent_node_id + layer_version + layer_ordinal` 唯一。
3. 同一 `parent_node_id + display_name + layer_version` 不允许完全重复。

索引：

1. `uidx_tree_nodes_session_public_node_number`
2. `idx_tree_nodes_parent_node_id`
3. `idx_tree_nodes_child_group_id`
4. `idx_tree_nodes_session_depth`
5. `idx_tree_nodes_superseded_at`

| `branch_count` | `integer` | 本轮计划生成的分支数 |
| `status` | `text` | `queued` / `reasoning` / `generating` / `completed` / `failed` |
| `input_source` | `text` | `voice` / `text` / `canvas` |
| `user_intent_text` | `text` | Orchestrator 归一后的标准意图文本 |
| `raw_input_text` | `text nullable` | 原始转写或文字输入，便于调试 |
| `design_intent_summary` | `text nullable` | 本轮意图摘要 |
| `assistant_reply` | `text nullable` | 对话区简短回复 |
| `error_message` | `text nullable` | 错误信息 |
| `created_at` | `timestamptz` | 创建时间 |
| `updated_at` | `timestamptz` | 更新时间 |

1. `idx_generation_tasks_session_created_at`
2. `idx_generation_tasks_session_status`

用途：

1. 表示一次生成 turn 中单个分支 brief 的执行状态。

关键字段：

| 字段 | 类型 | 说明 |

| `generation_task_id` | `uuid` | 所属生成任务 |

| `status` | `text` | `queued` / `generating` / `completed` / `failed` |
| `brief_payload` | `jsonb` | 保存 Brainstorm Assistant 输出的方向 brief |
| `image_url` | `text nullable` | 生成图 URL |
| `persisted_node_id` | `uuid nullable` | 最终落树节点 |
| `error_message` | `text nullable` | 错误信息 |
| `created_at` | `timestamptz` | 创建时间 |
| `updated_at` | `timestamptz` | 更新时间 |

索引：

1. `idx_branch_tasks_generation_task_id`
2. `idx_branch_tasks_generation_task_status`

## 4.6 tree_operations

用途：

1. 记录所有树结构变更。
2. 为 `refresh`、`delete`、`undo` 和 `redo` 提供事实来源。

关键字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | `uuid` | 主键 |
| `session_id` | `uuid` | 所属会话 |
| `task_id` | `uuid nullable` | 所属生成任务；delete/undo/redo 可为空 |
| `type` | `text` | `diverge` / `refresh` / `delete` / `undo` / `redo` |
| `target_node_id` | `uuid` | 目标节点 |
| `target_layer_version` | `integer nullable` | 目标层版本 |
| `affected_child_group_id` | `uuid nullable` | 本次影响的生成组 |
| `inserted_node_ids` | `jsonb` | 新插入节点 ID 数组 |
| `deleted_node_ids` | `jsonb` | 删除操作影响的节点 ID 数组 |
| `superseded_node_ids` | `jsonb` | 被隐藏或替代的节点 ID 数组 |
| `restored_node_ids` | `jsonb` | 撤回或重做时恢复的节点 ID 数组 |
| `undo_of_operation_id` | `uuid nullable` | 撤回对应的原始操作 |
| `redo_of_operation_id` | `uuid nullable` | 重做对应的原始操作 |
| `payload` | `jsonb` | 额外上下文，如 `branch_count`、reasoning 摘要 |
| `created_at` | `timestamptz` | 创建时间 |

索引：

1. `idx_tree_operations_session_created_at`
2. `idx_tree_operations_task_id`
3. `idx_tree_operations_target_node_id`
4. `idx_tree_operations_undo_of_operation_id`
5. `idx_tree_operations_redo_of_operation_id`

## 5. JSONB 字段策略

以下字段建议使用 JSONB：

1. `tree_nodes.voice_aliases`
2. `tree_nodes.form_language`
3. `branch_tasks.brief_payload`
4. `tree_operations.inserted_node_ids`
5. `tree_operations.deleted_node_ids`
6. `tree_operations.superseded_node_ids`
7. `tree_operations.restored_node_ids`
8. `tree_operations.payload`

原因：

1. 这些字段天然是数组或结构化对象。
2. v1 阶段不需要在 SQL 层对这些字段做复杂统计查询。
3. 使用 JSONB 可以减少表数量，保持 migration 简洁。

## 6. 关键写入语义

## 6.1 diverge

1. 在目标节点下插入新的子节点组。
2. 新节点共享同一个 `child_group_id`。
3. 记录一条 `tree_operations`，`type = diverge`，`inserted_node_ids` 为新节点集合。

## 6.2 refresh

1. 找到目标节点下最近一次生成组。
2. 生成新的 `child_group_id` 和新节点集合。
3. 旧生成组节点不删除，仅写入 `superseded_at` 和 `superseded_by_operation_id`。
4. `tree_operations.superseded_node_ids` 记录旧组节点集合，`inserted_node_ids` 记录新组节点集合。

## 6.3 delete

1. 根节点不可删除。
2. 删除目标节点及其整棵子树。
3. 节点不物理删除，而是标记 superseded。
4. `tree_operations.deleted_node_ids` 记录整棵子树节点集合。

## 6.4 undo

1. 撤回最近一次可撤回的 `diverge / refresh / delete`。
2. 若撤回 `diverge`，则隐藏其 `inserted_node_ids`。
3. 若撤回 `refresh`，则隐藏新组并恢复旧组 `superseded_node_ids`。
4. 若撤回 `delete`，则恢复 `deleted_node_ids`。
5. 撤回本身写入一条 `tree_operations`，`type = undo`。

## 6.5 redo

1. 只恢复最近一次 `undo` 指向的原始操作。
2. 若重做 `diverge`，恢复其插入节点。
3. 若重做 `refresh`，重新隐藏旧组并恢复新组。
4. 若重做 `delete`，再次隐藏删除节点。
5. 一旦执行新的 `diverge / refresh / delete`，此前 redo 语义失效。
6. 重做本身写入一条 `tree_operations`，`type = redo`。

## 6.6 chat

1. 闲聊、帮助、状态查询和节点解释不改变树结构。
2. 用户输入写入 `messages`，`role = user`，`kind = intent` 或 `transcript`。
3. Chat Assistant 回复写入 `messages`，`role = assistant`，`kind = chat` 或 `node_explanation`。
4. 不创建 `generation_tasks`。
5. 不创建 `branch_tasks`。
6. 不写入 `tree_operations`。
7. 不修改 `tree_nodes`。
8. 可更新 `sessions.last_mentioned_node_id`，前提是回复明确引用了某个节点；不得更新 `current_selected_node_id`。

## 6.7 memory_summary

1. 当对话超过 6 轮后，Orchestrator 调用 Memory Summarizer。
2. 每次触发时，Orchestrator 传入最近 6 轮对话和可选上一条 `memory_summary`。
3. 6 轮以内不调用 Memory Summarizer，只使用裁剪后的 `conversationHistory`。
4. Memory Summarizer 输出稳定偏好、有效约束、已否定方向、开放问题和短摘要。
5. 输出写入 `messages`，`role = system`，`kind = memory_summary`。
6. 不创建 `generation_tasks`。
7. 不创建 `branch_tasks`。
8. 不写入 `tree_operations`。
9. 不修改 `tree_nodes`。
10. 后续构造 Brainstorm Assistant 或 Chat Assistant 输入时，Orchestrator 可读取最近一条或少量 `memory_summary`。

## 7. 事务边界

以下写操作必须使用事务：

## 7.1 生成任务完成事务

在生成任务完成时，以下内容必须同事务提交：

1. 更新 `generation_tasks.status`
2. 更新 `branch_tasks`
3. 插入 `tree_nodes`
4. 插入 `messages`
5. 插入 `tree_operations`
6. 更新 `sessions.current_selected_node_id`、`last_executed_target_node_id`、`last_mentioned_node_id`、`next_public_node_number`

## 7.2 delete 事务

删除时必须同事务提交：

1. 校验目标节点存在且不是 root。
2. 收集目标节点整棵子树。
3. 标记相关节点 superseded。
4. 写入 `tree_operations`。
5. 更新 session 当前选中节点。
6. 写入操作摘要消息。

## 7.3 undo / redo 事务

撤回或重做时必须同事务提交：

1. 读取目标 `tree_operation`。
2. 更新相关节点可见性。
3. 写入新的 `undo` 或 `redo` 操作记录。
4. 写入系统摘要消息。
5. 更新 session 当前选中节点和 redo 语义。

## 7.4 chat 事务

闲聊分支可以使用轻量事务，保证用户消息和助手回复成对写入：

1. 插入用户输入消息。
2. 插入助手回复消息。
3. 必要时更新 `sessions.last_mentioned_node_id`。

该事务不得包含任何 `generation_tasks`、`branch_tasks`、`tree_nodes` 或 `tree_operations` 写入。

## 7.5 memory_summary 事务

记忆摘要写入是轻量事务：

1. 读取近期消息和上一条 `memory_summary`。
2. 插入新的 `memory_summary` 消息。
3. 必要时更新 `sessions.updated_at`。

该事务不得包含任何 `generation_tasks`、`branch_tasks`、`tree_nodes` 或 `tree_operations` 写入。

## 8. 持久层职责拆分

建议按对象拆 repository，而不是做一个大而全的 DAO：

1. `session-repository`
2. `message-repository`
3. `tree-node-repository`
4. `generation-task-repository`
5. `branch-task-repository`
6. `tree-operation-repository`

## 8.1 session-repository

1. 创建会话。
2. 更新当前选中节点。
3. 分配下一个 `public_node_number`。
4. 更新最近执行目标和最近提及节点。

## 8.2 message-repository

1. 插入用户标准意图。
2. 插入系统状态。
3. 插入 AI 回复和操作摘要。
4. 插入闲聊、帮助、状态查询和节点解释回复。
5. 插入和读取 `memory_summary`。
6. 按时间查询会话消息。

## 8.3 tree-node-repository

1. 插入节点。
2. 查询当前可见节点。
3. 查询目标节点及祖先路径。
4. 查询目标节点下最近一次生成组。
5. 标记节点 superseded。
6. 撤回或重做时恢复节点。

## 8.4 generation-task-repository

1. 创建生成任务。
2. 更新任务状态。
3. 查询会话中正在运行的生成任务。
4. 拒绝为 `chat / delete / undo / redo` 创建生成任务。

## 8.5 branch-task-repository

1. 批量创建分支任务。
2. 更新单个分支任务状态。
3. 记录图片 URL 和落树节点 ID。
4. 查询部分失败情况。

## 8.6 tree-operation-repository

1. 写入树操作日志。
2. 查询最近一次可撤回操作。
3. 查询最近一次 undo 对应的 redo 目标。
4. 查询某任务对应的树操作。
5. 提供 `refresh` 所需的生成组上下文。

## 9. 初版 migration 范围建议

第一版 migration 只需要覆盖：

1. `sessions`
2. `messages`
3. `tree_nodes`
4. `generation_tasks`
5. `branch_tasks`
6. `tree_operations`
7. 关键唯一索引和外键

以下内容可以暂缓：

1. `image_assets`
2. `node_reviews`
3. 复杂统计索引
4. 全文搜索

## 10. 与 shared schema 的对齐原则

数据库层字段命名应尽量与 shared schema 对齐：

1. 共享 schema 是 API 和业务层主语义。
2. 数据库字段允许使用下划线命名，例如 `public_node_number`。
3. repository 层负责在数据库行结构和共享 schema 之间转换。

需要特别对齐的字段：

1. `displayName` <-> `display_name`
