# 工业设计语音脑暴工具设计稿

日期：2026-06-12

## 1. 概述

本产品是一款面向企业工业设计师的 Web 端语音脑暴画布，用于在项目 kickoff 之后，把模糊需求快速发散成多个造型语言差异明显的概念方向。

产品核心不是“说一句出一张图”，而是“说一句，长出一棵概念树”。用户通过语音描述需求、补充限制、选定方向，AI 在白色画布上持续生成分支化的草图节点，帮助设计师在前期概念探索阶段更快打开思路，并保留完整的演化路径。

第一阶段核心用户为企业工业设计师。后续可扩展到共创 workshop 场景，但不纳入本次 MVP 范围。

## 1.1 当前统一口径（2026-06-16）

本文件中凡与下列规则冲突的旧描述，均以后述规则为准：

1. 主链路不再要求高风险树操作先确认后执行，所有有效语音输入默认直达执行。
2. 当前 MVP 的核心语音动作为：
   - `diverge`：沿当前节点新增一组子节点
   - `refresh`：刷新当前节点下最近一次生成出来的那组子节点
   - `delete`：删除目标节点及其整棵子树
   - `undo`：撤回最近一次 `diverge / refresh / delete`
   - `redo`：恢复最近一次被 `undo` 的操作
3. 本期先不把“纯节点选择命令”作为独立语音能力纳入实现范围。
4. 当前节点规则统一为：
   - 首轮默认根节点
   - 若本轮语音明确提到节点，则本轮使用该节点
   - 若本轮未提节点，则沿用 `currentSelectedNodeId`
5. `refresh` 若未明确数量，则沿用被刷新的那一组原始 `branchCount`；`diverge` 若未明确数量，则默认生成 `3` 个子方向。
6. 根节点不可删除；删除当前节点后，当前节点回退到父节点，没有父节点时回根节点。
7. 任意生成任务进行中时，输入发送按钮置灰，用户无法继续发送消息。

## 2. 目标与非目标

### 2.1 目标

1. 帮助工业设计师在 kickoff 阶段快速完成前期发散。
2. 让需求到图像方向的转换更快、更自然，减少手动整理和反复描述成本。
3. 通过树状结构保留方案的分支关系，让讨论过程更有结构。
4. 让 AI 输出不仅有图，还能解释每个方向为什么成立。

### 2.2 非目标

1. 不做高保真工业设计渲染或最终效果图工具。
2. 不做专业建模、CAD 或生产链路集成。
3. 不做多人协作和 workshop 联机能力。
4. 当前 MVP 不做任何依赖鼠标的节点点选、拖拽、框选、自由排版、框选编组、批量批注。
5. 当前 MVP 不做键盘文字输入作为主要创作路径，用户创作输入以语音为唯一主入口。
6. 不做大量参数滑杆和复杂控制面板，避免退化为通用生图工具。

## 3. 核心用户与使用场景

### 3.1 核心用户

企业内的工业设计师，尤其是在新项目启动、方向探索、前期概念发散阶段需要快速生成多个造型方向的人。

### 3.2 核心场景

用户在项目 kickoff 之后，针对一个仍然模糊的产品需求，通过语音输入描述目标、场景、风格倾向和功能重点。AI 基于这些输入生成多个造型语言差异明显的概念草图分支，用户选择其中一个方向继续深入或刷新当前层结果，逐步形成一棵概念演化树。

### 3.3 核心痛点

1. 前期发散速度慢，从语言到图像的转换成本高。
2. 设计讨论容易散，方案很多但缺少结构化关系。

## 4. 产品定位

一句话定位：

一个面向企业工业设计师的语音脑暴画布，让模糊需求沿着可视化分支快速长成多个造型方向。

差异化表述：

它不是“说一句出一张图”的 AI 生图工具，而是“说一句，长出一棵概念树”的工业设计前期发散工具。

## 5. 核心体验原则

1. 右边说，左边长：输入和理解发生在对话区，结果和演化发生在画布区。
2. 先发散，再收敛：前几轮草图更松、更强调方向差异，后续逐步更具体。
3. 结构先于数量：比起生成更多图，更重要的是用户始终知道当前位于哪条分支、这一轮发生了什么。
4. 图和解释并重：每个方向不仅有草图，也有简短解释支持判断。
5. 工业设计语境优先：提示、解释和方向命名都要贴近工业设计表达，而不是泛视觉创作语言。

## 6. 信息架构

### 6.1 整体布局

页面采用双区结构：

1. 左侧为白色无限画布，承载树状概念演化结构。
2. 右侧为轻量对话框，承载聊天记录、语音转写结果、AI 回复、输入提示和当前操作状态。

### 6.2 画布区

画布区是产品主空间，用于展示从根节点开始不断生长的概念树。

1. 根节点表示初始需求输入。
2. 每一层节点表示 AI 基于某轮上下文生成的一组方向。
3. 节点之间以分支关系连接，形成清晰的演化路径。
4. 每个节点都必须有稳定可读的命名，便于语音引用和系统复述。
5. 用户通过语音选中任一节点作为下一轮操作对象。

### 6.3 对话区

对话区不是参数面板，而是思考对话空间，主要包括：

1. 用户语音转写后的聊天记录。
2. AI 的简短理解与反馈。
3. 当前轮次的输入提示，例如建议从比例、材质、部件、品牌气质等维度继续描述。
4. 当前操作状态提示，包括本轮识别到的意图类型和作用对象。

当前 MVP 的对话区应遵循“默认简洁、运行时展开、完成后折叠”的原则：

1. 工作台进入后不展示冗长历史，默认只保留当前节点、最近摘要和主要操作入口。
2. 当系统处于转写、理解、确认、生成等运行态时，实时展开系统反馈历史，帮助用户理解当前发生了什么。
3. 当本轮结束后，系统反馈自动折叠为一条摘要，例如“已基于 12 号节点生成 3 个更轻薄的子方向”。
4. 高风险确认卡片优先级高于普通系统消息，出现时应始终保持可见。

### 6.4 页面模块

当前 MVP 工作台固定由以下模块组成：

1. `TopBar`：显示会话标题、连接状态和当前任务是否运行中。
2. `CanvasWorkspace`：左侧主空间，展示概念树、当前选中节点和最近一次新增结果高亮。
3. `CurrentTargetBanner`：右侧顶部，始终显示当前操作对象的名称、数字序号 ID 和层内位置。
4. `ConversationPanel`：右侧中部，展示用户转写、运行中系统反馈和完成后的摘要消息。
5. `RecordingBar`：右侧底部主操作区，包含长按录音按钮、波形/计时和上传状态。
6. `IntentStatusCard`：在系统理解阶段出现，展示识别到的操作类型、目标节点和预计结果。
7. `IntentStatusCard`：在系统理解或执行阶段出现，展示识别到的操作类型、目标节点和预计结果。

## 7. 节点模型

### 7.1 节点默认展示

每个节点默认显示：

1. 一张草图感概念图。
2. 一个简短且稳定的方向名称。
3. 一个节点独有的数字序号 ID，例如 `12`，用于语音引用和系统复述。
4. 一个用于当前层理解的层内序号，例如“当前层第 2 个方向”。

### 7.2 节点展开信息

用户选中或展开节点后，显示一段简短解释，同时覆盖以下三类内容：

1. 造型语言：例如圆润、硬朗、几何化、工具感、家居化。
2. 回应需求：说明该方向主要回应了哪些功能、场景或气质诉求。
3. 灵感来源：说明可能借鉴的产品语义、品类特征或视觉启发。
4. 节点名称和当前在树中的相对位置，便于用户继续语音引用。

解释长度应控制在短段落级别，避免节点信息过重。

## 8. 核心交互模型

### 8.1 基本循环

产品的基本工作流为：

说需求 -> 生成多分支 -> 阅读解释 -> 选择路径 -> 再说 -> 树继续生长

每一轮都应让用户清楚知道：

1. 当前选中了哪个节点。
2. 本轮输入作用于哪一层。
3. 本轮是新增分支、刷新当前层，还是继续下钻。

当前 MVP 采用“直接进入工作台”的单会话入口：

1. 用户进入产品后直接落到当前 `session` 工作台。
2. 不单独设计首页跳转，不把启动动作拆成额外的落地页或模式选择页。
3. 首次进入时右侧以 `RecordingBar` 引导用户直接说出第一个需求。

### 8.1.1 纯语音无鼠标操作模型

当前 MVP 只支持语音操作，不依赖鼠标或键盘完成创作流程。系统需要覆盖以下纯语音闭环：

1. 语音创建：用户通过语音描述初始需求，系统创建根节点和首层分支。
2. 语音浏览：用户可说“读一下当前层”“展开第二个方向”“回到上一层”等命令获取结构状态。
3. 语音删除：用户可说“删除这个节点”“删掉 12 号节点”，系统删除该节点及其整棵子树。
4. 语音撤销：用户可说“撤销上一轮”，MVP 支持撤回最近一次 `diverge / refresh / delete`。
5. 语音重做：用户可说“重做上一轮”，系统恢复最近一次被撤回的树操作。
6. 语音纠错：用户可说“不是这个，是左边第二个”“我刚才说的是更圆润，不是更家电化”。
7. 语音状态查询：用户可说“现在选中的是谁”“这一轮准备做什么”“还有哪些方向”。

系统不要求用户记忆内部 nodeId，但必须向用户稳定暴露可说出的名称、数字序号 ID 和层内序号。

### 8.2 三类核心语音指令

系统需要支持并识别以下三类核心操作意图：

#### A. 扩充上次描述，增加分支

用户希望在保留当前描述框架的基础上，再增加更多平级方案。

示例：

“再多给我几个更极简、更家电化的方向。”

系统结果：

在当前目标节点下新增更多同级分支，不覆盖已有结果。

#### B. 补充描述，刷新上次结果

用户希望在当前层增加新的限制或偏好，但分支数量保持不变。

示例：

“保留这组数量，但整体更轻薄一点、更适合办公环境。”

系统结果：

重生成当前层结果，保留分支数量，不新增层级。

#### C. 选定一个优化方向，继续发散

用户希望沿着某个已选方向继续深入，生成下一层分支。

示例：

“沿着第二个方向继续，做三个更柔和的子方向。”

系统结果：

在所选节点下新增下一级分支，进入更细化的方向探索。

### 8.3 意图识别策略

第一版不要求用户手动切换模式。系统应优先根据用户语句和当前选中节点自动判断意图类型。

为降低误判成本，系统需要在对话区清晰显示：

1. 已识别的操作类型。
2. 作用对象节点。
3. 本轮将执行的结果形态，例如“新增 3 个同级方向”或“刷新当前 4 个分支”。

如果用户发现理解错误，应能直接继续用自然语言纠正。

### 8.4 高风险操作确认机制

涉及树结构变更的高风险操作在执行前必须经过“复述改写 + 用户确认”两步：

1. Orchestrator 和大模型先将用户原话改写成一条清晰执行意图。
2. 系统在对话区和语音播报中复述，例如“我将基于‘轻薄办公感’方向刷新当前 4 个分支，并保持数量不变”。
3. 用户明确说出“确认”“继续”“是的”后才真正写树。
4. 若用户说“取消”“不对”“换一个”，本轮保持只读，不写入树。

固定规则表如下：

| 操作类型 | 是否必须确认 | 规则说明 |
|---|---|---|
| `refresh_layer` | 是 | 会生成当前层新版本，影响当前可见结果 |
| `branch_deeper` | 是 | 会在目标节点下新增子层 |
| `expand_branches` 作用于当前已明确选中节点 | 否 | 默认可直接执行 |
| `expand_branches` 作用于非当前节点或存在歧义 | 是 | 先确认目标节点，再执行 |
| 查询类操作 | 否 | 如“读一下当前层”“现在选中谁” |
| 纠错类操作 | 否 | 如“不是这个，是 12 号节点” |
| 单次撤销 | 是 | 为避免误撤销，先复述“将撤销最近一次已确认操作” |

MVP 不支持用户比较新旧版本，也不暴露版本对比视图。`refresh_layer` 的历史版本只用于内部回放和单次撤销。

### 8.5 关键页面流程

#### 8.5.1 首轮生成流程

1. 用户进入工作台后，通过 `RecordingBar` 说出第一轮需求。
2. 系统进入 `transcribing`，对话区显示用户语音转写结果。
3. 系统进入 `reasoning`，`IntentStatusCard` 可展示识别到的操作类型和预计结果。
4. 若本轮不需要确认，则直接进入 `generating`。
5. 左侧生成根节点和首层分支，右侧展示一条完成摘要。
6. 本轮结束后，运行中系统反馈自动折叠，只保留简要结果。

#### 8.5.2 高风险确认流程

1. 用户发出 `refresh_layer`、`branch_deeper` 或其他需确认的高风险命令。
2. 系统先完成转写和理解，再生成一条清晰的执行意图改写。
3. `ConfirmationCard` 出现，并通过语音播报请用户确认。
4. 用户说“确认”“继续”“是的”后，系统才进入 `generating` 并写树。
5. 用户说“取消”“不对”后，本轮结束，不写入树，只保留一条取消摘要。

#### 8.5.3 单次撤销流程

1. 用户说“撤销上一轮”。
2. 系统先检查最近一次是否存在可撤销的已确认树操作。
3. 若存在，则通过 `ConfirmationCard` 复述“将撤销最近一次已确认操作”。
4. 用户确认后，系统回滚该次 `TreeOperation`，并更新左侧树状态。
5. 撤销完成后，对话区仅保留一条撤销完成摘要。

#### 8.5.4 节点选中流程

1. 用户可通过数字序号、节点名称或层内序号选中目标节点。
2. 系统优先按数字序号解析，再尝试名称匹配，最后尝试层内序号匹配。
3. 若名称存在歧义，则进入澄清或确认流程，不直接更新目标节点。
4. 节点选中成功后，`CurrentTargetBanner` 立即更新。
5. 节点选中只改变当前 session 上下文，不直接写树。

## 9. 输出风格节奏

草图输出应随分支深度逐步变化：

1. 前几轮偏松散、概念化、方向差异明显。
2. 中间轮次开始呈现更清楚的轮廓关系和工业设计特征。
3. 后续深入节点可逐步变得更具体，便于团队继续讨论和筛选。

第一版不追求高保真效果图，应优先保证方向差异和概念表达清晰。

## 10. 提示系统

当用户不确定如何继续描述时，系统应提供少量发散提示。提示内容应贴近工业设计思维，优先从以下维度引导：

1. 比例关系。
2. 材质与 CMF 倾向。
3. 关键部件和交互区域。
4. 品牌气质与产品语义。
5. 使用场景与目标人群感受。

提示应短小、可直接接话，不应变成长篇教程。

## 11. MVP 功能范围

### 11.1 必做

1. Web 端白色脑暴画布。
2. 右侧对话框与语音输入转写。
3. 基于节点的多分支结果生成。
4. 节点草图与简短解释展示。
5. 每个节点稳定命名，并支持语音引用。
6. 三类核心意图的自动识别与状态展示。
7. 纯语音选中、浏览、确认、纠错、单次撤销能力。
8. 高风险树操作的复述确认机制。
9. 少量发散提示能力。

### 11.2 不做

1. 多人协作。
2. 高保真渲染与精细后处理。
3. 专业建模软件衔接。
4. 深度白板编辑能力。
5. 大量参数化控制。
6. 视觉 review 主链路能力，MVP 阶段先不纳入必做。

## 12. 端到端用户流程

1. 用户进入页面，看到左侧白色画布和右侧对话框。
2. 系统提示用户描述项目需求、目标用户、场景和期望气质。
3. 用户通过语音输入 kickoff 阶段的初始需求。
4. 系统完成语音转写，并在对话区显示理解摘要。
5. AI 在根节点下生成 3 到 5 个第一层方向，每个方向包含草图和解释。
6. 用户横向比较当前层结果。
7. 用户选择继续增加分支、刷新当前层，或沿某个方向继续向下发散。
8. 系统识别本轮意图，在对话区显示操作类型和作用对象。
9. 若操作涉及树结构变更，系统先复述改写后的执行意图并等待语音确认。
10. 用户确认后，画布更新，树结构继续生长。
11. 用户可随时通过语音撤销最近一次已确认操作、改选节点或要求系统读出当前层状态。
12. 用户重复该循环，直到得到足够清晰的概念方向。

## 13. 成功标准

第一版产品概念验证阶段，建议以以下标准判断是否成立：

1. 用户能在短时间内生成多个风格差异明显的初始方向。
2. 用户能理解并接受树状分支作为主要工作方式。
3. 用户能清楚分辨“增加分支”“刷新当前层”“继续下钻”三类操作。
4. 节点解释确实帮助用户做选择，而不是成为额外噪音。
5. 用户认为它比通用文生图工具更适合工业设计 kickoff 发散。

## 14. 后续扩展方向

不属于 MVP，但可作为后续路线：

1. workshop 多人共创模式。
2. 节点对比、收藏和筛选机制。
3. 方向汇总为方案板或提案页。
4. 更强的工业设计语义模板与行业化提示。
5. 与下游建模或设计交付工具的连接。
