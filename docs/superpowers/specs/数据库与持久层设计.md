## 1. 目标

本文档定义 MVP 阶段的数据库结构、关键索引、事务边界和持久层职责，重点回答：

1. 哪些核心对象需要落表。
2. 哪些字段必须结构化，哪些字段可以暂时用 JSONB 承载。
3. `refresh_layer`、`branch_deeper`、单次撤销等关键写树行为如何被持久化。
4. 持久层应该按什么边界拆 repository。

本文档当前属于设计与 DDL 草案，不代表已经完成数据库 migration 与业务实现。

## 2. 总体原则

1. 以 `PostgreSQL` 为主存储。
2. 以关系型字段承载查询主路径，以 `JSONB` 承载可变结构。
3. 所有会修改树结构的操作都必须形成 `TreeOperation` 记录。
4. 任何会影响多个节点和任务状态的写操作，必须在同一个事务里完成。
5. `refresh_layer` 不物理覆盖旧节点，而是通过版本化层和 superseded 标记实现。
6. MVP 只支持撤销最近一次已确认树操作，因此不设计 redo 队列。

## 3. 表清单

MVP 阶段建议最少落以下 6 张核心表：

1. `sessions`
2. `messages`
3. `tree_nodes`
4. `generation_tasks`
5. `branch_tasks`
6. `tree_operations`

后续扩展表：

1. `image_assets`
2. `node_reviews`

MVP 当前不要求先落后续扩展表；只需在设计中保留接口与字段预期。

## 4. 表设计

### 4.1 sessions

用途：

1. 表示一个完整脑暴会话。
2. 持有当前活跃节点、待确认节点和节点编号分配游标。

关键字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | `uuid` | 主键 |
| `title` | `text` | 会话标题 |
| `goal` | `text` | 会话目标摘要 |
| `product_domain` | `text` | 固定为 `industrial_design` |
| `status` | `text` | `active` / `archived` |
| `root_node_id` | `uuid nullable` | 根节点 |
| `active_node_id` | `uuid nullable` | 当前活跃节点 |
| `pending_node_id` | `uuid nullable` | 当前待确认节点 |
| `last_mentioned_node_id` | `uuid nullable` | 最近一次提及节点 |
| `next_public_node_number` | `integer` | 下一个可分配数字序号 |
| `created_at` | `timestamptz` | 创建时间 |
| `updated_at` | `timestamptz` | 更新时间 |

约束：

1. `next_public_node_number >= 1`
2. `product_domain = 'industrial_design'`

索引：

1. `idx_sessions_status`
2. `idx_sessions_updated_at`

### 4.2 messages

用途：

1. 保存用户转写、系统状态、AI 回复、确认摘要等右侧会话区消息。

关键字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | `uuid` | 主键 |
| `session_id` | `uuid` | 所属会话 |
| `task_id` | `uuid nullable` | 关联任务 |
| `role` | `text` | `user` / `assistant` / `system` |
| `kind` | `text` | `transcript` / `status` / `summary` / `confirmation` / `hint` |

索引：

1. `idx_messages_session_created_at`

用途：

1. 保存概念树节点。
2. 支持稳定命名、数字序号引用、层版本替换和单次撤销。

关键字段：

|---|---|---|

| `session_id` | `uuid` | 所属会话 |

| `created_from_task_id` | `uuid nullable` | 来源任务 |
| `depth` | `integer` | 树深度 |
| `layer_ordinal` | `integer` | 当前层稳定序号 |
| `layer_version` | `integer` | 当前层版本号 |
| `public_node_number` | `integer` | 会话内唯一数字序号 |
| `display_name` | `text` | 口头可说出的稳定名称 |

## 5. JSONB 字段策略

以下字段建议使用 `JSONB`：

1. `tree_nodes.voice_aliases`
2. `tree_nodes.form_language`
3. `tree_nodes.user_need_response`
4. `tree_nodes.inspiration_hints`
5. `branch_tasks.brief_payload`
6. `tree_operations.inserted_node_ids`
7. `tree_operations.superseded_node_ids`
8. `tree_operations.restored_node_ids`
9. `tree_operations.payload`

原因：

1. 这些字段天然是数组或结构化对象。
2. MVP 阶段不需要在 SQL 层对这些字段做复杂统计查询。
3. 使用 `JSONB` 可以减少表数量，保持 migration 简洁。

## 6. 关键写入语义

### 6.1 expand_branches

1. 保持目标节点与现有层版本不变。
2. 在目标节点下插入新的平级子节点。
3. 记录一条 `tree_operations`，`inserted_node_ids` 为新节点集合。

### 6.2 branch_deeper

1. 在目标节点下创建下一层子节点。
2. 不修改原有同层节点。
3. 记录一条 `tree_operations`，`inserted_node_ids` 为新子节点集合。

### 6.3 refresh_layer

1. 找到目标父节点下当前可见层节点。
2. 生成新的 `layer_version = old + 1` 节点集合。
3. 旧节点不删除，仅写入 `superseded_at` 和 `superseded_by_operation_id`。
4. `tree_operations.superseded_node_ids` 记录旧节点集合，`inserted_node_ids` 记录新节点集合。

### 6.4 undo

1. 只允许撤销最近一次已确认 `tree_operation`。
2. 若最近一次操作是 `expand_branches` 或 `branch_deeper`，则撤销时将 `inserted_node_ids` 对应节点标记为失效。
3. 若最近一次操作是 `refresh_layer`，则撤销时恢复 `superseded_node_ids`，并让新版本节点失效。
4. 撤销本身也写入一条 `tree_operations`，类型为 `undo`。

## 7. 事务边界

以下写操作必须使用事务：

### 7.1 turn 完成写入事务

在 `GenerationTask` 完成时，以下内容必须同事务提交：

1. 更新 `generation_tasks.status`
2. 更新 `branch_tasks`
3. 插入 `tree_nodes`
4. 插入 `messages`
5. 插入 `tree_operations`
6. 更新 `sessions.active_node_id`、`last_mentioned_node_id`、`next_public_node_number`

### 7.2 确认型操作事务

在用户确认高风险操作后，以下内容必须同事务提交：

1. 更新 `generation_tasks.confirmation_status`
2. 执行对应写树逻辑
3. 写入系统确认消息
4. 写入 `tree_operations`

### 7.3 单次撤销事务

撤销时必须同事务提交：

1. 读取最近一条可撤销 `tree_operation`
2. 更新相关节点有效状态
3. 写入新的 `undo` 操作记录
4. 写入系统撤销摘要消息

## 8. 持久层职责拆分

建议按对象拆 repository，而不是做一个大而全的 DAO：

1. `session-repository`
2. `message-repository`
3. `tree-node-repository`
4. `generation-task-repository`
5. `branch-task-repository`
6. `tree-operation-repository`

建议职责如下：

### 8.1 session-repository

1. 创建会话
2. 更新当前活跃节点
3. 分配下一个 `public_node_number`

### 8.2 message-repository

1. 插入用户转写
2. 插入系统状态
3. 按时间查询会话消息

### 8.3 tree-node-repository

1. 插入节点
2. 查询当前层可见节点
3. 查询目标节点及祖先路径
4. 标记节点 superseded
5. 撤销时恢复节点

### 8.4 generation-task-repository

1. 创建 turn 任务
2. 更新任务状态
3. 查询待确认任务

### 8.5 branch-task-repository

1. 批量创建分支任务
2. 更新单个分支任务状态
3. 查询部分失败情况

### 8.6 tree-operation-repository

1. 写入树操作日志
2. 查询最近一次可撤销操作
3. 提供 `refresh_layer` 所需的 superseded / restored 上下文

## 9. 初版 migration 范围建议

第一版 migration 只需要覆盖：

1. `sessions`
2. `messages`
3. `tree_nodes`
4. `generation_tasks`
5. `branch_tasks`
6. `tree_operations`
7. 关键唯一索引和外键

以下内容可以暂缓：

1. `image_assets`
2. `node_reviews`
3. 复杂统计索引
4. 全文搜索

## 10. 与 shared schema 的对齐原则

数据库层字段命名应尽量与 `packages/shared/src/schemas.ts` 对齐：

1. 共享 schema 是 API 和业务层主语义。
2. 数据库字段允许使用下划线命名，例如 `public_node_number`。
3. repository 层负责在数据库行结构和共享 schema 之间转换。

需要特别对齐的字段：

1. `displayName` <-> `display_name`
2. `publicNodeNumber` <-> `public_node_number`
3. `layerVersion` <-> `layer_version`
4. `rewrittenIntentForConfirmation` <-> `rewritten_intent_for_confirmation`
5. `supersededNodeIds` <-> `superseded_node_ids`
6. `restoredNodeIds` <-> `restored_node_ids`
