## 1. 目标

本文档描述第一版 MVP 的技术实现方案，重点覆盖：

1. 前端工作台结构。
2. 后端服务与编排层。
3. Agent 架构与职责边界。
4. 核心数据结构。
5. 一次完整生成链路的状态流。
6. 纯语音无鼠标操作模型、容错与树操作机制。

本文档不展开具体任务拆解和排期，主要回答“系统应该怎么搭”和“关键难点在哪里”。

## 1.1 当前统一口径（2026-06-16）

以下口径覆盖并更新本文档中仍残留的旧确认链路表述，后续实现和验收以本节为准：

1. 当前主链路采用“无确认直执行”模型，不再把 `awaiting_confirmation`、`ConfirmationCard`、语音 `confirm/cancel` 作为主产品流程的一部分。
2. 所有语音输入统一进入后端 `Turn Planner`，先做意图识别，再分流到“树操作执行器”或“生成执行器”。
3. 当前 MVP 先只支持五类核心意图：
   - `diverge`
   - `refresh`
   - `delete`
   - `undo`
   - `redo`
4. 本期先不支持“纯节点选择命令”作为独立语音操作；当前节点默认沿用 session 中的 `currentSelectedNodeId`，若语音中明确指到节点，则本轮用该节点作为目标节点。
5. `refresh` 的语义固定为：刷新“当前节点下最近一次生成出来的那组子节点”，而不是刷新当前层所有可见节点。
6. `delete` 的语义固定为：删除目标节点及其整棵子树；根节点不可删除。
7. `undo` / `redo` 采用标准操作栈语义：
   - `undo` 可撤回最近一次 `diverge / refresh / delete`
   - `redo` 可恢复最近一次被 `undo` 的操作
   - 一旦用户执行新的 `diverge / refresh / delete`，原有 `redo` 栈立即清空
8. 同一 session 任意生成任务运行中时，发送入口禁用，用户不能继续发送新消息，也不能插入 `undo / redo / delete`。
9. Session 中需要显式区分两个状态：
   - `currentSelectedNodeId`：当前默认操作节点
   - `lastExecutedTargetNodeId`：最近一次真正改树操作使用的目标节点
10. 每次生成都必须带有“生成组”概念，例如 `childGroupId`，用于支撑“刷新最近一次生成组”和 `undo / redo` 的精确恢复。

## 2. 总体架构

第一版采用 `Web 前端 + 后端 orchestrator + 多模型 agent` 的结构。

核心原则：

1. 前端只负责交互、展示和状态反馈，不直接承担复杂推理。
2. 后端 orchestrator 是唯一的流程调度者和树状态写入者。
3. Agent 只负责各自专长，不直接修改全局状态。
4. 所有生成结果都先结构化，再落到树节点中。
5. 当前 MVP 只支持语音操作，不依赖鼠标或键盘完成创作输入。

## 3. 角色划分

### 3.1 前端

前端负责：

1. 白色无限画布展示概念树。
2. 右侧对话框展示聊天记录、状态和提示。
3. 长按语音输入。
4. 语音节点选中、展开、继续发散、撤销和确认等操作触发。
5. 展示每轮处理中间状态。

前端不负责：

1. 意图识别。
2. 提示词展开。
3. 图像生成。
4. 树结构业务判断。
5. 依赖鼠标的节点点选和拖拽作为主交互路径。

### 3.2 后端 Orchestrator

Orchestrator 是核心后端服务，负责：

1. 接收前端请求。
2. 管理会话、树、节点和历史状态。
3. 调用语音转写服务。
4. 收集上下文并调用脑暴助理。
5. 将脑暴助理输出路由给生图助理。
6. 校验 agent 输出是否合法。
7. 写入节点、分支和聊天记录。
8. 将最终结果回传前端。

这是系统里唯一可以改树状态的角色。

### 3.3 脑暴助理

脑暴助理负责：

1. 理解本轮用户语音输入。
2. 判断本轮属于哪类操作。
3. 结合当前节点路径和上下文，扩写为结构化设计意图。
4. 拆出多个差异化分支 brief。
5. 输出面向用户的简短回复文案。

它不直接出图，不直接写树。

### 3.4 生图助理

生图助理负责：

1. 将结构化 brief 转换为图像生成提示词。
2. 保持统一的工业设计早期草图风格。
3. 确保同轮不同分支在造型语言上有足够差异。
4. 调用图像模型生成草图。

它不解释用户意图，不维护树状态。

### 3.5 视觉审稿助理

视觉审稿助理负责：

1. 对生成图进行视觉理解。
2. 判断结果是否符合 brief 和本轮设计意图。
3. 评估是否存在品类漂移、造型偏差或分支同质化。
4. 输出诊断和下一步优化建议。

它不是传统 OCR 角色，而是视觉评审角色。若底层模型支持 OCR，可将其作为图像理解的一部分能力。

## 4. 推荐技术栈

### 4.1 前端

建议：

1. `Next.js` 或 `React + Vite`
2. `TypeScript`
3. `React Flow` 用于树状画布
4. `Zustand` 维护前端 UI 状态
5. 浏览器原生 `MediaRecorder` 处理长按录音

原因：

1. React Flow 很适合节点和边的基础交互。
2. 前端类型化有助于和后端 schema 对齐。
3. MediaRecorder 足够支撑第一版长按录音。
4. `Zustand` 更适合当前 MVP，轻量且足以承载工作台 UI 状态、任务轮询结果和确认卡片状态。

### 4.1.1 纯语音交互约束

当前 MVP 的前端工作台应遵循以下约束：

1. 用户创作输入只通过语音进入系统，不提供键盘文本输入框作为主入口。
2. 画布默认仅作为展示与状态反馈空间，不要求用户通过鼠标点击节点完成主链路。
3. 节点必须暴露稳定的语音引用信息，包括节点名称、层内序号和父子关系提示。
4. 前端需要支持语音播报和可视化展示并行，以便用户同时通过听觉和视觉确认系统理解。

### 4.1.2 页面模块划分

当前 MVP 前端建议固定为以下模块：

1. `TopBar`：显示会话标题、连接状态和当前任务运行状态。
2. `CanvasWorkspace`：左侧主空间，展示概念树、当前选中节点和最近生成高亮。
3. `CurrentTargetBanner`：右侧顶部，始终显示当前目标节点的名称、数字序号和层内位置。
4. `ConversationPanel`：右侧中部，展示用户转写、系统反馈和完成摘要。
5. `RecordingBar`：右侧底部主操作区，承载录音按钮、波形/计时和上传状态。
6. `IntentStatusCard`：reasoning 阶段展示识别到的操作类型、目标节点和预计结果。
7. `IntentStatusCard`：在系统理解或执行阶段展示识别到的操作类型、目标节点和预计结果。

### 4.1.3 前端状态展示策略

前端应采用“默认简洁、运行时展开、完成后折叠”的展示策略：

1. 默认不展开冗长系统历史，只展示当前节点、最近摘要和主操作入口。
2. 当任务进入 `transcribing`、`reasoning`、`generating` 时，`ConversationPanel` 展开本轮反馈。
3. 当任务进入 `completed` 或 `cancelled` 时，本轮反馈折叠为一条摘要消息。
4. 运行态状态卡的可见优先级高于普通消息流。

### 4.2 后端

建议：

1. `Node.js + TypeScript`
2. `Fastify`
3. `PostgreSQL` 存储树、节点、消息和生成记录
4. `Redis` 做任务状态缓存和队列辅助
5. `BullMQ` 或等价任务队列处理异步生成流程
6. 轻量 `query builder`，固定采用 `Drizzle`

原因：

1. TypeScript 前后端统一，schema 更容易复用。
2. `Fastify` 更适合当前 MVP，链路直接、抽象较轻，便于快速打通 session、task、tree 和 provider orchestration。
3. 图像生成天然适合异步任务和队列化执行。
4. PostgreSQL 能较稳定地承载树节点、消息和生成记录。
5. 轻量 `query builder` 更适合当前“树状态 + 操作日志 + 单次撤销”这类可控性要求更高的数据写入场景，当前固定采用 `Drizzle`。

### 4.2.1 本地开发方式

第一版本地开发固定采用“应用本地运行 + PostgreSQL / Redis 使用 Docker”的方式：

1. `apps/web`、`apps/server`、worker 进程在本机直接运行，优先保证热更新和调试效率。
2. `PostgreSQL` 与 `Redis` 通过 Docker 启动，优先保证环境一致性。
3. 第一版不要求把全部应用都放入 Docker 容器中运行，避免增加本地开发阻力。

### 4.3 模型层

建议按能力拆分：

1. `ASR 模型`：负责音频转写。
2. `LLM`：负责脑暴助理。
3. `图像生成模型`：负责草图生成。
4. `VLM`：后续扩展时可负责视觉审稿。

第一版优先保证链路打通，不强绑定某个固定模型提供方。模型接入应通过适配层封装。

### 4.3.1 MVP 固定模型供应商方案

为降低第一版开发复杂度，MVP 固定采用单一主供应商 `硅基流动（SiliconFlow）`。

结合当前 MVP 的产品需求，第一版推荐固定为以下组合：

1. `ASR`：`FunAudioLLM/SenseVoiceSmall`
2. `LLM`：`deepseek-ai/DeepSeek-V4-Flash`
3. `图像生成`：`Tongyi-MAI/Z-Image-Turbo`
4. `VLM`：当前 MVP 不启用；后续若恢复视觉 review，再单独评估 `Qwen/Qwen3-Omni-30B-A3B-Instruct` 一类多模态模型

这样做的目标是：

1. 统一鉴权、计费与 API 入口
2. 降低多供应商编排复杂度
3. 让 prompt、日志和失败处理更集中
4. 在保证首版效果的前提下优先控制响应延迟和调用成本

推荐理由如下：

1. `FunAudioLLM/SenseVoiceSmall` 已出现在硅基流动官方语音转文本接口示例中，接入路径最直接，适合作为第一版默认 ASR。
2. `deepseek-ai/DeepSeek-V4-Flash` 更适合当前“语音理解 -> 意图改写 -> 结构化输出 -> 高风险复述确认”这条链路，重点在中文理解、结构化输出、低延迟和成本平衡。
3. `Tongyi-MAI/Z-Image-Turbo` 适合当前“首轮多分支快速出图”的任务形态，相比更重的高质量模型更有利于控制首版等待时间。
4. 视觉 review 已移出 MVP 主链路，因此当前不引入额外 VLM，避免增加编排复杂度和成本。

备选策略如下：

1. 若 `deepseek-ai/DeepSeek-V4-Flash` 在中文结构化输出上不稳定，可评估 `Pro/zai-org/GLM-4.7` 作为高稳定性备选。
2. 若 `Tongyi-MAI/Z-Image-Turbo` 的草图质量不足，可切换到 `Tongyi-MAI/Z-Image`，用更高生成质量换取更高延迟。
3. 若后续希望把音频直接送入多模态模型统一理解，可评估 `Qwen/Qwen3-Omni-30B-A3B-Instruct`，但这不属于首版 MVP 路线。

尽管模型供应商先固定，代码层仍建议保留 `agent-gateway` 适配边界，便于后续替换。

## 5. 前端实现方案

## 5.1 页面结构

页面采用双栏布局：

1. 左侧 `CanvasWorkspace`
2. 右侧 `ConversationPanel`

### 5.1.1 CanvasWorkspace

职责：

1. 展示根节点、分支节点和连线。
2. 展示当前语音选中的节点状态。
3. 支持节点展开查看解释和历史版本信息。
4. 展示节点状态，例如生成中、生成成功、待确认、失败。

建议子组件：

1. `CanvasViewport`
2. `BrainstormNodeCard`
3. `TreeEdge`
4. `NodeDetailPopover`

### 5.1.2 ConversationPanel

职责：

1. 展示聊天记录。
2. 展示当前选中节点信息。
3. 提供长按语音输入按钮。
4. 展示系统识别到的本轮操作。
5. 展示复述改写后的执行意图与确认状态。
6. 展示输入提示和优化建议。

建议子组件：

1. `MessageList`
2. `RecordingButton`
3. `CurrentTargetBanner`
4. `IntentStatusCard`
5. `ConfirmationCard`
6. `PromptHintCard`

## 5.2 前端状态

前端应维护两类状态：

1. `服务端主状态镜像`
2. `本地 UI 交互状态`

服务端主状态镜像包括：

1. 当前会话信息
2. 树节点列表
3. 当前选中节点
4. 消息记录
5. 当前任务状态
6. 当前待确认操作

本地 UI 状态包括：

1. 是否正在录音
2. 录音开始时间
3. 画布缩放和平移
4. 当前打开的节点详情
5. 临时错误提示
6. 最近一次系统语音复述内容

## 5.2.1 纯语音无鼠标操作模型

第一版前端必须支持以下语音工作流：

1. `create`：用户描述需求，触发首轮方向生成。
2. `browse`：用户请求系统读出当前层、当前节点、上一层或某个方向摘要。
3. `select`：用户通过节点名称、独有数字序号 ID、层内序号或相对表达选择目标节点。
4. `confirm`：用户确认高风险树结构变更。
5. `repair`：用户纠正系统识别结果，例如改目标节点、改动作类型、改约束词。
6. `undo`：用户撤销最近一次已确认树操作。

前端不要求用户知道内部 `nodeId`。前端负责将语音选择结果映射为结构化目标，并展示给用户确认。

## 5.3 长按录音交互

第一版采用明确输入边界：

1. 用户按下录音按钮开始录音。
2. 按住期间界面显示“正在对 AI 说话”。
3. 松手后结束录音并上传音频。
4. 系统进入 `transcribing -> reasoning -> awaiting_confirmation -> generating -> completed` 的任务状态流。

这可以有效避免环境讨论污染上下文。

## 5.4 节点命名与语音寻址

每个节点都必须有可供用户直接说出的名称。建议同时维护三类引用信息：

1. `displayName`：用户可理解的方向名称，例如“轻薄办公感”。
2. `publicNodeNumber`：会话内独有的数字序号 ID，例如 `12`。
3. `layerOrdinal`：当前层中的稳定序号，例如“第 2 个方向”。
4. `voiceAliases`：同义表达，例如“第二个”“12 号”“办公感那个”“刚才那个轻薄的”。

节点命名策略在开发前固定如下：

1. `displayName` 由 Brainstorm Assistant 生成，Orchestrator 只做长度和重复校验，不二次改写语义。
2. `displayName` 建议限制在 4 到 12 个中文字符或等价短语长度。
3. 同一父节点下不允许出现完全重复的 `displayName`；若冲突，要求模型重命名。
4. `publicNodeNumber` 由服务端按会话内单调递增分配，创建后不复用、不变更。
5. 语音播报优先使用“名称 + 数字序号”双重表达，例如“轻薄办公感，12 号节点”。

语音寻址的优先顺序建议为：

1. 明确数字序号 ID 命中
2. 明确节点名称命中
3. 当前层稳定序号命中
4. 最近一次提及节点命中
5. 若存在多个候选，则进入二次确认而不是直接落树

## 6. 后端实现方案

## 6.1 服务分层

建议拆为以下模块：

1. `api`
2. `session-service`
3. `tree-service`
4. `audio-service`
5. `orchestrator-service`
6. `agent-gateway`
7. `generation-worker`
8. `review-worker`
9. `persistence-layer`
10. `voice-control-service`

### 6.1.1 API

负责：

1. 接收会话创建请求
2. 接收音频上传
3. 接收节点选中和画布操作
4. 查询树状态和消息历史

### 6.1.2 Session Service

负责：

1. 创建会话
2. 管理会话级元数据
3. 维护当前活跃节点
4. 维护最近一次已确认节点和最近一次待确认节点

### 6.1.3 Tree Service

负责：

1. 创建根节点
2. 增加同级分支
3. 刷新当前层
4. 向某个节点增加子分支
5. 管理节点状态和层级关系

### 6.1.4 Orchestrator Service

负责一次完整请求的编排：

1. 取上下文
2. 调脑暴助理
3. 校验输出
4. 派发生图任务
5. 收集结果
6. 写库
7. 返回前端

### 6.1.6 Voice Control Service

负责：

1. 管理纯语音交互的状态机
2. 解析确认、取消、撤销、查询型命令
3. 将语音引用解析为候选节点集合
4. 为高风险操作生成复述文案
5. 处理二次确认回合

### 6.1.5 Agent Gateway

负责封装模型调用，隔离上层业务与模型提供方。

至少提供：

1. `transcribeAudio`
2. `runBrainstormAssistant`
3. `generateSketch`
4. `reviewGeneratedImage`

## 6.2 异步任务流

生图和审稿不建议塞进同步 HTTP 请求链路。更合理的方式是：

1. API 接收音频后创建任务。
2. 后端立即返回任务 ID。
3. Worker 异步执行转写、推理、生图。
4. 若后续启用视觉 review，则作为异步补充任务执行，不阻塞主链路。
5. 前端通过轮询或 websocket 订阅状态更新。

建议任务状态枚举：

1. `queued`
2. `transcribing`
3. `reasoning`
4. `generating`
5. `awaiting_confirmation`
6. `completed`
7. `failed`

## 7. Agent 架构设计

## 7.1 调度原则

系统不是多个 agent 自主聊天，而是 `Orchestrator 主调度 + 专职 agent 执行`。

原则：

1. 只有 orchestrator 持有全局状态。
2. agent 输入尽量最小化，只给完成当前任务需要的上下文。
3. agent 输出必须结构化。
4. 任一 agent 返回异常时，orchestrator 可降级或重试。
5. 任何会修改树的高风险操作都必须经过复述确认后再执行。

## 7.2 脑暴助理输入

建议输入包含：

1. 当前用户转写文本
2. 当前选中节点 ID
3. 从根到当前节点的路径摘要
4. 当前层已有分支摘要
5. 本会话的产品目标和风格约束

## 7.3 脑暴助理输出

建议输出结构：

```ts
type BrainstormActionType =
  | "expand_branches"
  | "refresh_layer"
  | "branch_deeper";

type VisualDirectionBrief = {
  briefId: string;
  targetParentNodeId: string;
  publicNodeNumber?: number;
  displayName: string;
  label: string;
  intentSummary: string;
  formLanguage: string[];
  userNeedResponse: string[];
  inspirationHints: string[];
  variationAxis: string;
  promptIntent: string;
};

type BrainstormAssistantResult = {
  actionType: BrainstormActionType;
  targetNodeId: string;
  branchCount: number;
  designIntentSummary: string;
  assistantReply: string;
  confirmationRequired: boolean;
  rewrittenIntentForConfirmation?: string;
  promptHints: string[];

};
```

说明：

1. `actionType` 映射三类核心语音指令。
2. `directionBriefs` 是传给生图助理的中间层。
3. `assistantReply` 用于右侧对话框展示。

## 7.4 生图助理输入

建议输入包含：

1. 单个 `VisualDirectionBrief`
2. 会话级统一视觉风格约束
3. 当前深度层级
4. 可选的同层兄弟分支摘要

## 7.5 生图助理输出

```ts

  imageId: string;

```

说明：

1. `promptUsed` 便于后期调试。
2. `visualSummary` 可用于节点说明和审稿输入。

## 7.6 视觉审稿助理说明

视觉审稿助理不纳入当前 MVP 主链路，先作为后续扩展能力保留。

后续启用时建议输入包含：

1. 原始用户本轮输入
2. 当前目标节点摘要
3. 对应的 `VisualDirectionBrief`
4. 生成图片
5. 同层其余分支的摘要或图片引用

后续启用时建议输出：

```ts

  reviewId: string;

  score: number;
```

说明：

1. `score` 用于后续筛选或自动重试策略。
2. `nextStepSuggestions` 可直接反馈给前端作为继续发散提示。

## 8.6 TreeOperation

为支持 `refresh_layer` 回放、撤销和版本管理，建议新增树操作日志：

```ts
type TreeOperationType =
  | "expand_branches"
  | "refresh_layer"
  | "branch_deeper"
  | "undo";

  sessionId: string;
  generationTaskId?: string;

  targetNodeId: string;

  nextLayerVersion?: number;

  supersededNodeIds: string[];
  restoredNodeIds: string[];

};
```

`refresh` 的实现应基于“当前节点下最近一次生成组”的版本切换，而不是物理覆盖旧节点。旧节点保留，只是被标记为 superseded。

MVP 约束：

1. 支持撤回最近一次 `diverge / refresh / delete`。
2. 支持恢复最近一次被 `undo` 的树操作。
3. 不提供新旧版本对比界面。
4. 旧版本仅用于内部回放、撤回和重做。

## 9. 一次完整请求链路

用户从长按到节点生成的完整流程如下：

1. 前端用户长按录音按钮。
2. 前端采集音频，松手后上传到 `/sessions/:id/voice-turns`。
3. API 创建 `GenerationTask`，返回 `taskId`。
4. Worker 调用 `transcribeAudio` 得到文本。
5. Orchestrator 读取当前选中节点和路径上下文。
6. Orchestrator 调用脑暴助理，得到 `BrainstormAssistantResult`。
7. Orchestrator 校验 `actionType`、`targetNodeId`、`branchCount`。
8. 若属于高风险树操作，则生成复述文本并将任务置为 `awaiting_confirmation`。
9. 用户语音确认后，Orchestrator 为每个 `VisualDirectionBrief` 创建 `BranchTask`。
10. 生图助理返回 `GeneratedSketch`。
11. Tree Service 根据 `actionType` 和 `TreeOperation` 写入节点或刷新当前层版本。
12. Message 记录用户转写、AI 回复、确认结果和系统状态。
13. 前端收到任务完成通知，更新画布和聊天区。

## 11. 关键实现难点

### 11.1 树上下文裁剪

不能把整棵树直接塞给脑暴助理，否则成本高且容易混乱。需要稳定的上下文摘要策略：

1. 当前节点路径摘要。
2. 当前层兄弟节点摘要。
3. 本轮用户输入。

### 11.2 三类意图稳定识别

用户自然语言常常混合多个意图。第一版需要优先保证：

1. 操作类型识别准确。
2. 作用节点明确。
3. 分支数量受控。

### 11.2.1 语音容错与确认机制

高风险操作必须经过以下流程：

1. 原始转写文本保留入库。
2. 大模型输出结构化动作和复述改写文本。
3. 系统向用户展示并播报复述文本。
4. 用户明确确认后才执行写树。
5. 若用户纠正目标节点、数量或动作类型，则回到 `reasoning` 重新生成而不是直接套改旧结果。

固定规则表如下：

| 操作类型 | 是否必须确认 | 说明 |
|---|---|---|
| `refresh_layer` | 是 | 会生成当前层新版本 |
| `branch_deeper` | 是 | 会新增子层 |
| `expand_branches` 作用于当前 `activeNodeId` | 否 | 默认可直接执行 |
| `expand_branches` 作用于非当前节点或候选节点不唯一 | 是 | 先确认目标 |
| 查询类操作 | 否 | 不写树 |
| 纠错类操作 | 否 | 重新进入推理，不写树 |
| 单次撤销 | 是 | 避免误撤销 |

### 11.3 分支差异控制

系统不能只生成多个“看起来不同但其实很像”的结果。需要脑暴助理显式给每个分支指定不同的 `variationAxis`，生图助理按轴生成。

### 11.4 审稿结果可信度

视觉审稿助理后续启用时可能过度自信，因此更适合作为建议层，而不是绝对裁决层。

### 11.5 端到端延迟

一次完整链路包含：

1. 转写
2. LLM 推理
3. 多张图生成
4. 高风险确认往返

必须通过异步任务、状态可视化和合理的分支数控制，避免用户感觉系统卡住。

## 12. 标注规范建议

为保证评测可执行，第一版建议配套最小标注规范：

### 12.1 Brainstorm Intent Set

每条样本至少包含：

1. `session_goal`
2. `selected_node_summary`
3. `ancestor_path_summary`
4. `sibling_summaries`
5. `transcript_text`
6. `gold_action_type`
7. `gold_target_node_reference`
8. `gold_branch_count`
9. `requires_confirmation`

标注规则：

1. 动作类型只能标一个主标签。
2. 如果用户语句存在歧义但不足以安全执行，标为需要确认。
3. 目标节点优先标稳定名称，其次标层内序号，不直接暴露内部 `nodeId` 给标注员。

### 12.2 Brief Quality Set

每条样本至少包含：

1. `brief_id`
2. `variation_axis`
3. `display_name`
4. `intent_summary`
5. `form_language`
6. `human_usability_score`，1 到 5 分
7. `axis_distinctness_score`，1 到 5 分

评分锚点建议：

1. `1` 分：几乎不可执行或与其他分支同质
2. `3` 分：基本可执行，但差异或清晰度一般
3. `5` 分：方向清晰、可执行、与兄弟分支明显不同

### 12.3 Voice Safety Set

每条样本至少包含：

1. `transcript_text`
2. `predicted_action_type`
3. `risk_level`
4. `rewritten_confirmation_text`
5. `human_decision_should_confirm`
6. `human_decision_should_execute`

该数据集用于验证“哪些语句必须确认”而不是只验证最终分类。

## 13. Session Service 维护当前节点的策略

主选择权交给用户，但 Session Service 仍需要维护少量状态以保证语音链路稳定：

1. `activeNodeId`：最近一次被用户明确确认的当前节点。
2. `pendingNodeId`：系统候选但尚未被用户确认的节点。
3. `lastMentionedNodeId`：最近一次在对话中被系统复述或用户提及的节点。

建议策略：

1. 只有用户明确选择或确认后，`activeNodeId` 才更新。
2. 系统推测出的候选节点先写入 `pendingNodeId`，等待确认。
3. 若任务进行中用户切换节点，新任务读取新的 `activeNodeId`，旧任务不得在完成后覆盖该状态。
4. 若节点引用歧义，优先用 `pendingNodeId` 发起确认，而不是静默猜测。
5. 当用户说“继续刚才那个”时，先尝试 `lastMentionedNodeId`，若置信度不足则要求确认。
6. 当用户明确说出数字序号 ID 时，优先直接解析该节点，不再依赖 `lastMentionedNodeId`。

## 14. MVP 建议范围

第一版建议只支持：

1. 单用户单会话。
2. 长按语音输入。
3. 3 到 4 个分支生成。
4. 基础节点树展示。
5. 每个节点稳定命名和语音引用。
6. 节点解释展示。
7. 高风险操作确认与单次撤销。
8. 基础任务状态流。

第一版暂不支持：

1. 多人协作。
2. 自动无限重画。
3. 高级白板编辑。
4. 模型自治修树。
5. 下游建模工具连接。
6. 视觉 review 主链路。
7. redo 和新旧版本对比。
