# 无确认语音链路改造设计

日期：2026-06-15

## 1. 目标

将当前工作台从“所有输入先确认再执行”的链路，调整为“语音输入后直接执行”的链路。

本次改造的直接目标：

1. 删除确认卡、确认按钮、取消按钮在主产品链路中的作用。
2. 每一轮语音输入直接落到树状态，不再进入 `awaiting_confirmation`。
3. 若本轮描述未提节点，则沿用“上一轮实际执行时的目标节点”。
4. 若本轮描述未提数量，则默认生成 `3` 个方向。
5. 保留单次撤回最近一次已执行树操作。
6. 将测试文档改为新的 6 轮链路，并使自动化测试与文档一致。

## 2. 新产品规则

### 2.1 执行规则

1. 用户每次提交语音或转写文本后，系统直接解析并执行。
2. 不再出现“待确认任务”。
3. `confirm` / `cancel` 不再属于主产品链路。

### 2.2 目标节点规则

1. 如果本轮描述明确提到节点，则使用该节点。
2. 如果本轮描述没有提到节点，则沿用“上一轮实际执行时的目标节点”。
3. 如果当前会话还没有任何已执行树操作，则默认使用根节点。

“上一轮实际执行时的目标节点”定义为：

- 最近一次真正完成写树的操作所使用的 `targetNodeId`
- 不包含取消或未执行的操作
- 不包含单纯查询类输入

### 2.3 生成数量规则

1. 如果本轮描述明确提到数量，则使用描述中的数量。
2. 如果本轮描述没有提到数量，则默认生成 `3` 个方向。
3. 数量仍受系统最小值和最大值约束。

### 2.4 撤回规则

1. 仍只支持撤回最近一次已执行树操作。
2. 撤回不需要二次确认。
3. 撤回后树恢复到上一轮执行前的状态。

## 3. 目标链路

测试文档主链路改为以下 6 轮：

1. 首轮输入需求
   - 基于根节点
   - 若描述里提到数量则按描述，否则默认 3
2. 基于节点 2 进行下钻到第二层级扩展
   - 明确指定节点 2
   - 若未提数量则默认 3
3. 进行第二层级的节点刷新
   - 若描述未提节点，则沿用第 2 轮实际执行目标节点
   - 若未提数量则默认 3
4. 进行第一层级节点 3 的下钻扩展
   - 明确指定第一层级节点 3
   - 若未提数量则默认 3
5. 进行节点 3 的第二层级第 1 个节点扩展
   - 明确指定节点 3 下的第二层级第 1 个节点
   - 若未提数量则默认 3
6. 撤回
   - 撤回第 5 轮

## 4. 后端设计

### 4.1 Orchestrator

`processVoiceTurn` 改为：

1. 解析 transcript
2. 解析目标节点
3. 解析数量
4. 调用 agent 输出结构化结果
5. 直接持久化树节点
6. 直接返回已完成任务

不再执行：

1. 创建 `awaiting_confirmation` 任务
2. 写入 confirmation message
3. 等待 `confirmTask`
4. 等待 `cancelTask`

### 4.2 目标节点解析

目标节点解析顺序改为：

1. transcript 中的明确节点引用
2. 请求体显式传入的 `targetNodeId`
3. session 中记录的“上一轮实际执行目标节点”
4. 根节点

### 4.3 session 状态

当前建议将“上一轮实际执行目标节点”存放在 session 侧，优先复用：

- `activeNodeId`

改造后的语义：

- `activeNodeId` = 最近一次已执行树操作使用的目标节点

每次树操作真正执行完成后，统一更新：

1. `activeNodeId`
2. `lastMentionedNodeId`
3. `nextPublicNodeNumber`

### 4.4 任务状态

主链路任务状态简化为：

1. `queued`
2. `generating`
3. `completed`
4. `failed`

兼容期可以暂时保留旧字段，但主链路不再写入：

1. `awaiting_confirmation`
2. `confirmationRequired`
3. `confirmationStatus`
4. `rewrittenIntentForConfirmation`

### 4.5 confirm / cancel API

方案 2 下建议：

1. 从主产品链路移除调用
2. 路由先保留，避免大范围破坏
3. 测试文档中移除其作为主流程验收项

后续可在下一轮再决定是否彻底删除接口与 schema 字段。

## 5. Agent 设计

### 5.1 Mock Agent

需要调整：

1. 默认数量改为 `3`
2. 不再强制 `confirmationRequired=true`
3. `assistantReply` 改为直接执行摘要，而不是确认文案

### 5.2 SiliconFlow Prompt

需要调整：

1. 删除“所有输入都先确认后执行”的系统约束
2. 要求模型只输出直接可执行结果
3. 保留 `targetNodeId` 与 `branchCount` 的合法性约束
4. 若用户未提数量，则默认输出 `3`

## 6. 前端设计

### 6.1 Store

需要删除或降级：

1. `pendingAction`
2. `confirmPendingAction`
3. `cancelPendingAction`
4. 依赖 `awaiting_confirmation` 的 UI 状态

保留：

1. `selectedNodeId`
2. `currentTargetNodeId`
3. 与真实 API 同步的消息、树、任务状态

### 6.2 UI

需要移除主链路中的：

1. `ConfirmationCard`
2. 确认/取消按钮交互
3. “等待确认”状态文案

需要保留或加强：

1. 当前目标节点展示
2. 最新一轮执行摘要
3. 树状态实时刷新

### 6.3 语音选中

本次顺带修正：

1. `选择 2 号节点`
2. `选择“某个名称”`
3. `选择当前层第二个方向`

这些输入应只更新当前目标节点，不创建生成任务。

## 7. 测试文档改造

`docs/superpowers/specs/测试文档.md` 需要改为：

1. 删除所有确认/取消前置条件
2. 将主 E2E 链路改为 6 轮执行链路
3. 每轮都记录：
   - 输入文本
   - 实际使用的目标节点
   - 实际生成数量
   - 操作类型
   - 树状态变化
4. 单独注明默认规则：
   - 未提节点时沿用上一轮实际执行目标节点
   - 未提数量时默认生成 3 个

## 8. 验证策略

### 8.1 自动化

需要更新：

1. server API 测试
2. web 源码结构测试
3. 与确认链路相关的旧断言
4. 新的默认节点 / 默认数量规则测试

### 8.2 手工 E2E

需要按新文档跑 6 轮：

1. 首轮生成
2. 节点 2 下钻
3. 第二层刷新
4. 第一层节点 3 下钻
5. 节点 3 的二层第 1 个节点扩展
6. 撤回

每轮确认：

1. 输入文本
2. 目标节点
3. 生成数量
4. 树状态
5. 对话区摘要

## 9. 风险与边界

1. 当前 schema、数据库设计、技术文档中大量使用 confirmation 概念，本轮先做“主链路停用”，不强求一次性删净。
2. 若 `activeNodeId` 被重新定义为“最近一次实际执行目标节点”，需要检查是否影响已有前端选中逻辑。
3. 节点选择链路当前是错误行为，本轮一并修复，否则“默认沿用上一轮目标节点”无法稳定使用。
4. 旧测试里很多用例默认依赖 `awaiting_confirmation`，改造后会出现大面积测试更新。

## 10. 实施建议

建议按以下顺序实施：

1. 先改后端规则与 mock / agent 输出
2. 再改前端 store 和确认 UI
3. 再改测试文档
4. 最后补自动化测试并重新跑一遍 6 轮手工 E2E
